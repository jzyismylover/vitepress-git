app.use & 全局变量 #

为什么在一些时候需要使用 app.use() 的方式去注册插件，为什么有时候像 echarts 或者说 axios 就不需要呢 ??? 推荐阅读 讲解链接

  • 其实本质上我们使用 app.use 的时候是因为这些插件提供了对应的 install 方法，插件的运行依赖于 install 方法的一些操作才能正常运行。
  • 而像 echarts 或者 axios 这些他们导出的就是一个可以直接使用的实例，不需要依赖于 install 方法。对于这类插件我们如果需要额外配置的话就是像项目里面的 axios 一样抽成一个函数的形式，如果想简单使用的话其实就可以进行全局挂载(Vue3 是不推荐使用的)

async & await #

廖雪峰老师的介绍

async #

function getName() {
  return 'hello world'
} // hello world

async function getName() {
  return 'hello world'
}  // Promise { 'hello world' }

console.log(getName()) // 针对两个不同的函数是有不同的输出的

从上面的代码我们可以知道，普通函数和 async 函数是完全不一样的，普通函数的返回值取决于内部的 return，async 函数的返回值一定是一个 promise 对象，只是说会根据内部不同的状态返回对应的值而已。

⭐ 状态变换机制

  • 内部的 return 作为 then 函数接收的参数

    • 内部成功执行完成但无 return ---> Promise.state === 'resolve' -----> then 接受 undefined
    • 内部成功执行完成有 return ---> Promise.state === 'resolve' -------> then 接受 return 的值

  • 内部的 throw 直接作为 catch 捕获的参数

    • 内部抛出 ------> Promise.state === 'reject' ---------> catch 接受报错

可能看上去不是特别理解，那就举三个例子吧。

// 验证内部成功执行完成但无 return -> Promise.state === 'resolve' -> then接受undefined
async function getName() {
  console.log('wyl')
}

getName()
  .then(res => console.log(res)) // undefined
  .catch(err => console.log(err))

//验证内部成功执行完成有 return -> Promise.state === 'resolve' -> then接受 return的值
async function getName() {
  console.log('wyl')
  return '10'
}

getName()
  .then(res => console.log(res)) // '10'
  .catch(err => console.log(err))

// 验证内部抛出  --> Promise.state === 'reject' --> catch 接受报错
async function getName() {
  throw('早点休息！！！')
}

getName()
  .then(res => console.log(res))
  .catch(err => console.log(err))  // 早点休息！！！

await #

await 通常是配合 async 使用的，相比于 promise，我觉得它更加地直观一些。

const getData = () => {
  return new Promise((resolve, reject) => {
    // 进行数据请求的逻辑，完成后进行 resolve
    axios.get('/path').then(res => {
      // res 是使用 axios 得到的请求结果
      resolve(res)
    })
  })
}

// 普通的做法
function fetchData() {
  const promise = getData().then(res => {
    // 数据赋值
  })
}

// 使用 async / await 的做法
async function fetchData() {
  const res = await getData()
  // 数据赋值
}

🔑 其实对于上面的 await 我们可以把它看成是这样的形式

async function fetchData() {
  new Promise((resolve, reject) => {
    resolve(getData)
  }).then(res => {})// res 就是getData 的返回值
}

咋眼一看其实就是 promise。

🔑 await 还有一个特别重要的点就是它在一个作用域内会对下面的代码进行阻塞

async function fetchData() {
  const res = await 2
  console.log('wyl')
}

console.log('10')
fetchData()
console.log('jj')

// ??? 猜猜会输出什么
// 答案是 10 jj wyl

为什么不是输出 10 wyl jj 呢？其实这就要涉及到事件循环的概念了，这个概念可能不是三眼两语就能讲明白的，还是需要多去看一些文章博客然后手敲一些代码区理解，推荐看一下 事件循环(看浏览器的事件循环就好)，看完以后再看上面的输出就知道了。

  • 执行流程
  1. 执行 console.log('10') -- 主执行栈清空
  2. 执行 fetchData() -- fetchData 入栈
  3. 执行 fetchData() 内部代码，执行到 await 2 -- 将 const res = xxx 和后面的console.log 注册为微任务(xxx 其实就是 2)
  4. 执行 console.log('jj')
  5. 同步代码执行完成，去微任务队列取任务执行，const res = 2; console.log('wyl')
  6. 最后呈现的就是 10 jj wyl

正是因为内部的阻塞，所以我们才需要使用 await 在请求到远程的数据后再进行操作。总的来说 async 和 await 就是一种更加简洁异步书写方法而已，其实本质上和 promise 是类似的。

flex 布局 #

推荐阅读一下廖雪峰老师的flex布局

里面讲的其实都是挺详细的，那我就挑我觉得比较难以理解的地方吧。

  • flex-basis

    • flex-basis 是里面我一开始看的时候觉得比较难理解的一个地方， flex-basis 到底有什么用呢？？？从我的观点来看，flex-basis 其实就是一个 width 属性(主轴是 row)，它定义了一个子项目占据的空间，但它的优先级比 width 高，也就是设置了 flex-basis width 就会被覆盖。
    • 本质的区别就是 flex-grid 和 flex-shrink 会根据 flex-basis 的值来计算主轴是否有剩余的空间从而缩放，而 width 是没有这个特性的，只是说不设置 flex-basis 默认就有一个 auto 的值而已。
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta http-equiv="X-UA-Compatible" content="IE=edge">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Document</title>
  <style>
    .container {
      display: flex;
      flex-direction: row;
    }
    .item1 {
      /* 尽管 width写在 flex-basis 后面
      但是容器的宽度还是被定义为 300px */
      flex-basis: 300px;
      width: 100px;
      height: 100px;
      margin: 10px;
      background-color: yellow;
    }
    .item2 {
      width: 100px;
      height: 100px;
      margin: 10px;
      background-color: yellow;
    }
  </style>
</head>
<body>
  <div class="container">
    <div class="item item1"></div>
    <div class="item item2"></div>
  </div>
</body>
</html>

Vue3 globalProperties #

文档阅读

// 之前 (Vue 2.x)
Vue.prototype.$http = () => {}

// 之后 (Vue 3.x)
const app = createApp({})
app.config.globalProperties.$http = () => {}

!!! Vue2 的 main.js 和 Vue3 的 main.js 的区别

// vue2
import Vue from 'vue'
import App from './App.vue'
import Vue from './router'
import store from './store'
//引入Element-ui
import ElementUI from 'element-ui'

Vue.config.productionTip = false;
Vue.use(ElementUI)
Vue.prototype.$http = ()=>{
	console.log(xxx)
}

// vue2 是通过 new Vue({}) 实例一个对象的方式来生成实例
// 所以我们可以在 Vue 构造函数的原型上设置变量和属性达到全局共享的目的
// 为什么全局共享呢？可以理解为 main.js 里面的实例是父实例，组件的是子实例
// 子实例继承了父实例的属性、方法、原型
// 所以我们可以在组件中使用 this.$http 的形式访问变量
new Vue({
	router,
	store,
	render: (h) => h(App),
}).$mount('#app')


// vue3
import {createApp }from 'vue'
import App from './App.vue'
import Vue from './router'
import store from './store'
import ElementUI from 'element-ui'


// vue3 是通过 createApp 函数来创建实例
// 摈弃了原来构造函数的概念 -- 所以也就没有了原型的概念了
// 官网说createApp 返回一个提供应用上下文的应用实例。应用实例挂载的整个组件树`共享`同一个上下文
// 所以对于 vue2 的 prototype 可以使用其他的方式进行替代
// 也就是 app.config.globalProperties
const app = createApp(App)
app.config.globalProperties.$http = ()=>{
	console.log(xxx)
}
app.use(ElementUI )
app.use(store).use(router).mount('#app')

🔑 this.$echarts.init() 在一开始看来我觉得是错误的语法，但是其实是没有错的，因为 this 拿到的依然是 vue 的实例，只是说创建的方式不一样了而已。所以说只要我们能够拿到正确的 this，也是可以通过 this.$echarts.init 的方式来完成工作的。

// main.ts 里面挂载
app.config.globalProperties.wyl = '10'

<!-- 可以在 App.vue 里面测试一下 -->
<template>
  <div id="app">
    <router-view />
  </div>
</template>

<script>
import { defineComponent } from 'vue'
export default defineComponent({
  components: {},
  mounted() {
    console.log(this.wyl) // 拿到的是 '10' --> 所以 $echarts 那个也是类似的
  },
  setup() {
    // setup 是拿不到 this 的，所以不能通过上面这种方式去拿到全局挂载的变量
  }
})
</script>

vue3 获得全局挂载

<!-- 可以在 App.vue 里面测试一下 -->
<template>
  <div id="app">
    <router-view />
  </div>
</template>

<script>
import { defineComponent } from 'vue'
export default defineComponent({
  components: {},
  setup() {
    // 在 vue3 的 setup 中是通过这种方式去拿
    const { appContext } = getCurrentInstance() // 解构赋值
    const proxy = appContext.config.globalProperties
    console.log(proxy.wyl) // '10'
  }
})
</script>

Compositon Api & Option Api #

推荐阅读

option api

<script>
export default ({
  components: {},
  data() {},
  method: {},
  mounted() {}
})
</script>

composition api

<script>
import { defineComponent } from 'vue'
export default defineComponent({
  components: {},
  setup() {// setup 是在 options api 执行前执行的，所以
    // return 出去的数据可以被 options api 的一些周期拿到
    return {} 
  }
})
</script>

我觉得我们需要知道的是为什么 Vue3 改成了 composition api 去实现，相比于 options api 有哪些优点。

  • 逻辑组织
  • 逻辑复用
  • 更好的 typescript 支持

至于说如何去更好使用的话就只能多吸收多体会了。

Mock 工作机制 #

mock 其实本质上的工作流程就是在中途拦截了客户端到服务器的请求(得先有请求才能拦截)，然后根据一定的数据生成规则去模拟服务器给我们返回数据。mock 广泛用于很多领域，对于前端来说我们用的就是 Mock.js 这个库来实现 mock。
